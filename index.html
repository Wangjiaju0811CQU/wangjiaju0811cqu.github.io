L1 多策略切割接入差异分析

1. 目标

在 chapter1/text_splitter.py 中通过统一入口接入多种切割策略，并明确不同方法在工程接入、检索质量、资源成本上的差异，支持快速选型。

  1. build_text_splitter(chunk_size, chunk_overlap, strategy=...)

  3. export_split_files_for_strategies(...)

3. 方法差异总览（接入视角）

方法 接入复杂度 检索质量 上下文完整性 存储/索引成本 推荐场景
fixed_overlap 低 中 中 中 通用默认、快速上线
semantic_boundary 中 高（文本类） 高 中 说明文、FAQ、运营文档
heading_hierarchy 中 高（结构化文档） 高 中 Markdown/手册类内容
code_function 中-高 高（代码问答） 高（代码逻辑） 中 Python 代码库问答
parent_child 高 高 高 中-高 质量优先、需要父子上下文
late_chunking 中 中-高（长上下文模型） 很高 低-中（chunk 数少） 长上下文模型推理

4. 本项目当前结果（chunk 数）

基于当前 index.json：

  1. fixed_overlap: 47
  2. semantic_boundary: 48
  3. heading_hierarchy: 48
  4. code_function: 48
  5. parent_child: 47
  6. late_chunking: 35

结论：

  1. late_chunking 的 chunk 数明显更少，索引体积更小，但检索粒度更粗。
  2. 其余策略 chunk 数接近，差异主要体现在“chunk 边界质量”而不是数量。

5. 接入差异详解

5.1 fixed_overlap（默认基线）

优点：

  1. 接入最简单，风险最低。
  2. 对文档格式无要求。

缺点：

  1. 容易在语义中间切断。
  2. 多跳问题时命中稳定性一般。

工程建议：

  1. 先用它做 baseline。
  2. 线上保留它作为回退策略。

5.2 semantic_boundary（文本质量优先）

优点：

  1. 句子/段落边界更自然。
  2. 检索片段可读性更高。

缺点：

  1. 实现和调参比固定切割复杂。
  2. 不规则文本（表格、混排）收益有限。

工程建议：

  1. 中文说明文档优先试这个。
  2. 与 fixed_overlap 做 A/B，对比命中率与回答引用准确性。

5.3 heading_hierarchy（结构文档优先）

优点：

  1. 标题天然形成语义单元。
  2. metadata 可带标题层级，便于可解释检索。

缺点：

  1. 依赖文档标题结构质量。
  2. 非 Markdown 文档会回退到语义切割。

工程建议：

  1. 对知识库中的 .md 手册优先启用。
  2. 可配合 source_file + h1/h2 做过滤检索。

5.4 code_function（代码问答优先）

优点：

  1. 按函数/类切割，代码逻辑完整。
  2. 适合“某函数做什么、在哪改”的问答。

缺点：

  1. 目前主要针对 Python（AST）。
  2. 跨函数依赖仍可能分散到多个 chunk。

工程建议：

  1. 代码知识库单独用这个策略建库。
  2. 检索时优先展示 code_symbol metadata。

5.5 parent_child（质量与上下文平衡）

优点：

  1. 子块检索精度高，父块可补充上下文。
  2. 对复杂问答更稳。

缺点：

  1. 接入复杂度高于单层切割。
  2. metadata 设计要清晰（parent_id、chunk_level）。

工程建议：

  1. 检索命中 child，生成时补 parent 上下文。
  2. 适合核心业务知识库（质量优先）。

5.6 late_chunking（长上下文模型）

优点：

  1. 上下文保留最完整。
  2. chunk 数少，索引压力小。

缺点：

  1. 依赖长上下文模型能力。
  2. chunk 粒度较粗，精确定位能力下降。

工程建议：

  1. 仅在模型窗口足够大时使用。
  2. 可配合 reranker 弥补粗粒度问题。

9. 最终结论

这 6 种方法不是互斥关系，而是“按文档类型分层接入”更合理：

  1. 通用文本：semantic_boundary
  2. 结构文档：heading_hierarchy
  3. 代码文档：code_function
  4. 核心高质量检索：parent_child
  5. 长上下文模型：late_chunking
  6. 统一兜底：fixed_overlap

10.1 定量结果

10.2 关键效果差异结论

  1. 当前最明显差异来自 late_chunking
  • 向量数从 47/48 降到 35（约减少 25%），索引体积和检索候选数量更小。
  • 单 chunk 更大（平均 512），上下文完整性更好，但语义定位会变粗，容易把不相关内容一起带进来。
  1. semantic_boundary vs fixed_overlap
  • 数量和平均长度非常接近，当前数据集上差异主要体现在“切分边界更自然”，不是“规模变化”。
  • 对问答效果的提升通常体现在引用连贯性，而不是召回数量。
  1. heading_hierarchy、code_function 当前收益未显现
  • 这次输入主要是 pdf/docx/txt，缺少 Markdown 标题结构和 .py 代码语法结构。
  • 因此两者在当前样本上实际退化为与语义切分近似的结果。
  1. parent_child 当前表现接近基线
  • 目前导出为 child 粒度（include_parent=False），所以和 fixed_overlap 文本上接近。
  • 真正优势要在检索阶段“child 命中 + parent 回补”才能体现。

10.3 对 RAG 实际效果的影响（结合 ANN 检索）

  1. 小 chunk（~370）
  • 精准定位更好，Top-K 噪声更低。
  • 但上下文可能不完整，需要更多 chunk 拼接。
  1. 大 chunk（~500+）
  • 更容易保留完整段落和上下文。
  • 但 ANN 返回后可能夹带无关信息，生成阶段更依赖模型过滤能力。
  1. 在你这批数据上的推荐
  • 当前主策略：semantic_boundary
  • 需要更低索引成本且模型窗口足够大时：late_chunking
  • 若要发挥 parent_child：后续把检索链改成 parent 回补模式

L1 工作记忆层实现说明

目标

在 RAG 问答链之上增加一层会话工作记忆（L1），实现：

  1. 摘要 + Buffer 混合记忆策略
    保留最近 3-5 轮原始对话（当前实现为 4 轮），更早历史压缩为摘要。
  2. 维护结构化 task_state 字段（JSON）
    让模型快速感知“当前任务目标、进度、事实、待办”等状态。

本次改动文件

  1. l1_memory.py
    新增 L1 记忆核心实现（摘要更新、task_state 更新、会话管理）。
  2. qa_chain.py
    RAG Prompt 和 LCEL 链改造，支持传入 memory_context。
  3. app.py
    接入会话级记忆管理，并在 /ask 中完成“读记忆 -> 回答 -> 写记忆”闭环。
  4. templates/index.html
    前端自动生成并持久化 session_id，每次请求携带该 ID。

L1 结构

每个 session_id 对应一个独立记忆对象：

  1. running_summary
    压缩后的历史摘要（旧对话内容）。
  2. recent_turns
    最近原始对话缓冲区（当前为 4 轮，即 8 条 user/assistant 消息）。
  3. task_state
    结构化 JSON，字段如下：

请求流程（/ask）

  1. 读取 session_id 对应记忆对象。
  2. 组装 memory_context（摘要 + 最近对话 + task_state JSON）。
  3. 调用 RAG 链：
    • 输入：question + memory_context
    • 检索：文档 Top-K
    • 生成：受 L1 记忆与文档共同约束
  4. 将本轮 question/answer 回写 L1：
    • 超出 Buffer 的旧对话压缩进摘要
    • 自动更新 task_state
  5. 返回回答结果，同时附带最新 task_state（便于前端或调试使用）。

关键接口

  1. L1MemoryManager.get_or_create(session_id)
    获取或创建会话记忆对象。
  2. L1ConversationMemory.build_prompt_context()
    生成可注入 Prompt 的 L1 文本上下文。
  3. L1ConversationMemory.record_turn(user_message, assistant_message)
    记录一轮对话并触发摘要/状态更新。
  4. GET /memory/<session_id>
    可查看当前会话记忆快照（摘要、recent_turns、task_state）。

教学版实现拆解（可直接用于讲解）

1. 先讲整体架构

  1. RAG 负责“从知识库找答案”。
  2. L1 负责“把会话压缩成模型能快速理解的上下文”。
  3. 两者在 POST /ask 里合并：question + memory_context + 检索文档。

建议上课先画这个最小闭环：

用户问题 -> 读取会话记忆 -> 构建 memory_context -> RAG 检索+生成 -> 回写记忆

2. 再讲 L1 的三个核心数据结构

对应代码：l1_memory.py

  1. recent_turns：保留最近几轮原始对话（短期 Buffer）。
  2. running_summary：把溢出的旧对话压缩成摘要。
  3. task_state：结构化状态 JSON（目标、事实、待办、最新请求等）。

这三者拼成最终上下文，输出在 build_prompt_context。

3. 解释“每轮如何更新记忆”

入口函数：record_turn

每轮做 3 件事：

  1. 先把本轮 user/assistant 追加到 recent_turns。
  2. 如果超过窗口上限，把溢出部分交给 _update_summary 做压缩。
  3. 调用 _update_task_state 更新结构化状态。

4. 解释“摘要链”和“状态链”怎么做

  1. 摘要链：_build_summary_chain
    用 Prompt 约束 LLM 只输出简洁摘要，防止历史无限增长。
  2. 状态链：_build_state_chain
    用 Prompt 强制输出固定字段 JSON。
  3. 稳定性兜底：_safe_json_loads + _normalize_task_state
    防止模型输出脏 JSON 导致系统崩溃。

5. 解释 L1 在接口层如何接入

对应代码：app.py

  1. 根据 session_id 获取会话对象：get_or_create。
  2. 请求前读取 memory_context：memory.build_prompt_context()。
  3. 调用链路后写回记忆：memory.record_turn(...)。
  4. 响应返回最新 task_state，方便前端和调试观察。

6. 解释 Prompt 如何同时吃“记忆 + 检索”

对应代码：qa_chain.py

  1. context 由 retriever 从向量库取文档。
  2. memory_context 来自 L1。
  3. question 是当前问题。
  4. 三者一起送入 Prompt（见 build_prompt）。

7. 一句话教学总结

L1 不是替代 RAG，而是给 RAG 增加“会话连续性和上下文控长能力”，核心实现点就是 摘要 + 最近轮次 + task_state 三件套。

当前默认参数

  1. 最近窗口：max_recent_rounds = 4
  2. 摘要最大长度：max_summary_chars = 1200
  3. 会话上限：max_sessions = 200（超出会淘汰最早会话）

L1 Cache 基准测试结果（已提炼）

数据来源：chapter1/code_refactor/l1_cache_benchmark_result.json
测试脚本：chapter1/code_refactor/l1_cache_benchmark.py

关键指标（仅保留重点）

指标 数值 说明
测试轮数 12 覆盖短会话到长会话
全局压缩率 -11.99% 前期固定开销导致整体未降
第 9-12 轮压缩率 21.35% 长会话阶段开始明显提效
峰值单轮节省 519 字符（第12轮） 单轮节省 31.86%
最大上下文长度 baseline 1629 / L1 1110 L1 有效限制长度上限
L1 更新耗时 3.1575 ms/轮（均值） 维护摘要和 task_state 的额外成本

为什么这个指标会先下降

这里“下降”指全局压缩率为负（-11.99%），也就是短会话阶段 L1 反而更长。核心原因是固定成本：

  1. L1 每轮都会携带固定结构：历史摘要 标题、最近对话 标题、task_state JSON。
  2. 在前几轮对话里，baseline 只包含很短的历史文本；这时固定结构占比很高。
  3. max_recent_rounds=4 让最近 8 条消息原文保留，短会话阶段“压缩收益”还没出现，先看到的是“结构开销”。

为什么后面会优化

长会话时，baseline 与 L1 的增长方式不同：

  1. baseline（全量历史）是近似线性增长：轮数越多，拼接文本越长。
  2. L1 是“有上限”的增长：摘要上限 + 最近窗口 + task_state，整体趋于稳定。
  3. 当轮数到第 9 轮以后，baseline 增长超过固定结构开销，L1 开始反超并持续节省。
  4. 结果上可见：第 9-12 轮压缩率为 21.35%，第 12 轮单轮节省 31.86%。

简版结论

  1. L1 的主要价值是控制中长会话上下文膨胀，不是短会话立刻降成本。
  2. 提效拐点在第 9 轮左右，之后字符和 token 成本开始明显下降。
  3. 该结果支持继续演进 L2（Redis 近期记忆层）：L1 负责压缩，L2 负责近期高频检索。

可继续优化的方向

  1. 将 task_state 字段按业务域拆分（如：policy_topic、risk_level、pending_user_action）。
  2. 引入“摘要质量评估”与重写机制，避免摘要长期漂移。
  3. 结合数据库持久化 session_id -> L1，支持服务重启后记忆恢复。
